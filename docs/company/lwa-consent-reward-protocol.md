# LWA Consent and Reward Protocol

## Purpose
This document defines the safety boundary for future guild, mission, reward, and optional proof-of-contribution systems.

It exists to prevent the project from drifting into hidden compute, hidden AI training, deceptive mining, or illegal payout language.

## Core Rule

Nothing about user contribution, training, compute, or rewards may be hidden.

Everything must be:

- opt-in
- disclosed
- typed
- documented
- revocable where possible
- legally reviewed before payout-style systems

## 1. Consent Protocol

### Required consent flags

```text
opt_in_training
data_use_notice
task_consent
compute_consent
reward_terms_acknowledged
```

### Consent rules

- no default hidden participation
- no hidden resource use
- no vague “support the network” language that masks compute use
- no silent training use

## 2. Reward Protocol

### Allowed reward directions

```text
credits
badges
reputation
cosmetics
discounts
supporter_status
legally reviewed payouts later
```

### Not allowed without later review

- guaranteed payouts
- passive income claims
- investment language
- token appreciation claims
- securities-like promises

## 3. Guild and Mission Directions

Safe mission directions:

- Hook Battle
- Clip Rank Arena
- Caption Judge
- Thumbnail Forge
- Render QA
- creator testing loops

These must remain:

- transparent
- opt-in
- task-defined
- reward-defined

## 4. Optional Blockchain Bridge

Blockchain is optional only.

Allowed future use:

- proof of contribution
- badge verification
- collectible cosmetics
- off-chain-to-on-chain proof bridges after legal review

Not allowed:

- hidden mining
- hidden staking
- passive income framing
- undisclosed wallet requirements

## 5. Current Repo Alignment

Current repo alignment is conceptual only.

There are no verified live guild, compute-sharing, payout, or blockchain systems in the product today.

That means:

- do not claim them publicly
- do not imply them in sales copy
- keep all future work behind protocol, consent, and legal review

## 6. Testing Expectations

Before any consent or reward feature ships:

- verify explicit opt-in UI
- verify revocable state where appropriate
- verify terms acknowledgment
- verify no hidden background tasks
- verify no payout or income claims leak into copy
